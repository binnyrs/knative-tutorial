Getting Started

Overview

The getting started allows you to deploy Knative services using Knative Serving. This tutorial shows example of deploying a simple Greeter service in following languages,

  • Nodejs

  • Java

While deploying application in different languages demonstrates the polyglot capabilities as well as some key Knative concepts.

The completed solutions is available in the Solutions folder.

Environment

Its assumed that you have setup the minishift and deployed Knative serving. If not please vist setup section to do the same before proceeding further.

Configure the CLI

Before we go further with the build its necessary that we configure the command line to interact with Docker Daemon of the minishift and we use the right compatible version of the OpenShift CLI.

eval $(minishift docker-env) && eval $(minishift oc-env)

Sources

Clone the sources from https://github.com/redhat-developer-demos/knative-tutorial

git clone {repo-url}

For the rest of the document we call the cloned folder as $PROJECT_HOME.

Create a directory under $PROJECT_HOME called build, you will be using this directory as your working directory.

mkdir -p $PROJECT_HOME/build

You can always refer to the Solutions if you need help.

Points to Ponder

After you have deployed the Java service, you would have noticed that it has triggered a new Kubernetes deployment something like greeter-00002-deployment. Wondering the reason for it, lets compare the two service.yaml that were deployed:

Table 1. Greeter Service

NodeJs service

Java service

 
apiVersion: serving.knative.dev/v1alpha1
kind: Service
metadata:
  name: greeter
spec:
  runLatest:
    configuration:
      revisionTemplate:
        spec:
          container:
            image: dev.local/greeter:1.0.0
 
apiVersion: serving.knative.dev/v1alpha1
kind: Service
metadata:
  name: greeter
spec:
  runLatest:
    configuration:
      revisionTemplate:
        spec:
          container:
            image: dev.local/greeter:2.0.0

If you observe the only diffrence between the two service deployments is the version of the Docker Image, which is a configuration change. Knative follows 12-factor application principles where in every configuration change leads to a new revision,

kubectl get configurations.serving.knative.dev (1)
kubectl get revisions.serving.knative.dev (2)
1 Will return only one configuration called greeter
2 Will return two revisions something like greeter-00001 and greeter-00002

Each revision will trigger new deployments, and since the service is configured with runLatest it always serves the latest revision. Check Revisions and Configurations for more details on revisions, configurations and other possible revision rollout strategy.
