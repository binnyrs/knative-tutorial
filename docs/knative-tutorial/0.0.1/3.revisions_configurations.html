Revisions and Configurations

Overview

Knative follows 12-factor application principles where in every configuration change leads to a new revision. The configuration change could be image change, environment variables gets added or removed from the configuration.

kubectl get configurations.knative.dev (1)
kubectl get revisions.knative.dev (2)
1 Will return only one configuration called greeter
2 Will return two revisions something like greeter-00001 and greeter-00002

Rolling out changes

In the previous service deployments we did update the image from NodeJS to Java, now lets add a new environment variable(s) to the configuration and we will observe that it triggers new revision and deployment :

Adding Environment variable

cd $PROJECT_HOME/build/getstarted/java

Create a new service-env.yaml in $PROJECT_HOME/build/getstarted/java with the following content:

apiVersion: serving.knative.dev/v1alpha1
kind: Service
metadata:
  name: greeter
spec:
  runLatest:
    configuration:
      revisionTemplate:
        spec:
          container:
            env:
              - name: MESSAGE (1)
                value: Hello World
            image: dev.local/greeter:2.0.0
1 Updating the configuration with a environment variable called "MESSAGE"

The GreeterController.java is already setup to use the MESSAGE variable via @Value annotation:

package com.example.demo;

import org.springframework.beans.factory.annotation.Value;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;

/**
 * GreeterController
 */
@RestController
public class GreeterController {

    @Value("${MESSAGE:Knative on OpenShift}")
    private String message;

    @GetMapping("/")
    public String greet() {
        return"Java::"+message;
    }
}

Deploy Service

kubectl apply -f service-env.yaml

The service deployment should now trigger a new revision something like greeter-00003 and a new dpeloyment for the revision named greeter-00003-deployment.

Invoking Service should return a response like Hello World instead of the earlier Java::Knative on OpenShift

Rolling back changes

To Rollback Changes we can could always delete the application and redeploy the previous version. As Knative is backed by Istio we could still have the newer deployment in but without routing traffic to it.

Knative allows us to do this using a strategy called "Pinned Revisions"

Applying Pinned Revision

Create a new service-pinned.yaml in $PROJECT_HOME/build/java with the following content:

apiVersion: serving.knative.dev/v1alpha1
kind: Service
metadata:
  name: greeter
spec:
  pinned:
    revisionName: greeter-00001 (1)
    configuration:
      revisionTemplate:
        spec:
          container:
            env:
              - name: MESSAGE
                value: Hello World
            image: dev.local/greeter:2.0.0
1 In this case we are setting it to existing revision called greeter-0001

Though a new revision has been created with previous deployment, in the current service deployment we are keeping the configuration in tact as previous but updating the service route to a defined revision instead of latest.

You can set revisionName to any available revision. You can find all the available revisions using the following command:

kubectl get revisions.serving.knative.dev

Deploy Service

kubectl apply -f service-pinned.yaml

Invoke Service

The applications can be invoked only using knative-ingress-gateway which was deployed with Knative serving deployment.

The following snippet shows how to invoke the greeter nodejs service that was deployed earlier:

export IP_ADDRESS=$(kubectl get node -o 'jsonpath={.items[0].status.addresses[0].address}')

export SERVICE_NODEPORT=$(oc get svc knative-ingressgateway -n istio-system -o 'jsonpath={.spec.ports[?(@.port==80)].nodePort}')

export HOST_URL=$(kubectl get routes.serving.knative.dev greeter  -o jsonpath='{.status.domain}')

curl -H "Host: ${HOST_URL}" ${IP_ADDRESS}:${SERVICE_NODEPORT}

The curl command above should return a response like NodeJs::Knative on OpenShift
